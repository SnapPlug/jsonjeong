1. 머신 셋업 (평생 1회)

새 노트북/머신에서만 한 번. 이미 셋업된 환경이면 건너뜁니다.

# 패키지 매니저
brew install mise gh

# Docker Desktop (Kamal 배포용)
brew install --cask docker

# ~/.zshrc 에 mise 활성화 + PATH
cat >> ~/.zshrc <<'EOF'
eval "$(mise activate zsh)"
export PATH="$HOME/bin:$PATH"
EOF
source ~/.zshrc

# Ruby 3.3 + Rails 8
mise install ruby@3.3
mise use --global ruby@3.3
gem install rails -v "~> 8.1"

# GitHub 인증
gh auth login
Docker Desktop 은 첫 실행 시 동의 클릭 1회 필요. docker ps 가 에러 없이 빈 테이블을 출력하면 OK.

2. new-saas 스크립트 설치

스크립트 자체는 이 레포의 docs/new-saas 에 있습니다. ~/bin/ 에 복사:

설치 (한 번만)
mkdir -p ~/bin
cp /Users/jason/jason/docs/new-saas ~/bin/new-saas
chmod +x ~/bin/new-saas

# PATH 확인 (~/.zshrc 에 이미 추가했다면 source 만)
source ~/.zshrc
new-saas --help
스크립트는 자체 완결형. Rails / git / (옵션) gh CLI 만 있으면 동작합니다. 다른 머신으로 옮길 때 이 파일 하나만 복사하면 됩니다.

3. 스크립트 사용법

3-1. 인터랙티브 (권장)

cd ~/projects
new-saas snapdeck

스크립트가 5개 질문을 묻고 자동으로 다음을 수행:

  1. rails new snapdeck --css tailwind
  2. CLAUDE.md 생성 (maker/client 템플릿 분기)
  3. .gitignore 강화 (AI 캐시 무시)
  4. Rails 8 내장 인증 generator (선택)
  5. Kamal/Cloudflare 배포 템플릿 (config/deploy.yml + .kamal/secrets + DEPLOY.md)
  6. 첫 git 커밋
  7. GitHub 레포 생성 + push (선택)

3-2. 비대화형 (CI / 일괄 작업)

# 자체 메이커 프로젝트 (public)
new-saas snapdeck \
  --type maker \
  --lang en \
  --github-org SnapPlug

# 고객사 프로젝트 (private)
new-saas acme-portal \
  --type client \
  --lang ko \
  --private \
  --github-org acme-corp

# GitHub 레포 생성 없이 로컬만
new-saas experiment --no-github

# 인증 generator 건너뛰기
new-saas landing-page --no-auth

3-3. 옵션 레퍼런스

옵션설명Default
--typemaker 또는 client프롬프트
--langko / en / bilingual프롬프트
--github-orgGitHub 조직/사용자명프롬프트
--privatePrivate 레포maker=public, client=private
--publicPublic 레포—
--no-authRails 인증 generator 건너뛰기인증 포함
--no-githubGitHub 레포 생성 건너뛰기생성함
--no-deploy-templateKamal/Cloudflare 템플릿 건너뛰기포함함

4. CLAUDE.md 템플릿 비교

스크립트가 --type 에 따라 두 가지 CLAUDE.md 를 생성합니다.

🚀 Maker 템플릿

  • 언어 정책: 사이트/카피 영문 only, 대화 한글
  • 운영 사이클: 빌드 인 퍼블릭, OSMU, /go-viral-or-die
  • "하지 말 것": Devise, npm, Postgres, Service Object 남발
  • 스택 결정: Rails 8 + SQLite + Polar(MoR) 확정
  • 가시성: Public 기본

🏢 Client 템플릿

  • 클라이언트 정보 필드: 담당자/계약/컨택
  • 언어 정책: ko/en/bilingual 선택 (i18n 옵션)
  • 협업 규칙: Conventional Commits, PR 리뷰, 보호 브랜치
  • 인수 인계 체크리스트: README, ARCHITECTURE, .env.example, 보안 검토
  • 가시성: Private 기본
  • "하지 말 것": 클라이언트 데이터 로컬 보관, 무단 통합
두 템플릿 모두 Rails Omakase / Hotwire / Tailwind / Kamal 같은 기술 스택 부분은 동일. 다른 건 운영/협업 컨텍스트 뿐.

6. 배포 단계 (시간 있을 때)

MVP 코딩이 끝나고 배포 가능한 상태가 되면. 스크립트가 만든 DEPLOY.md 에 상세 가이드.

  1. 도메인 — Cloudflare Registrar 에서 구매
  2. Origin Cert — Cloudflare SSL/TLS → Origin Server → Create (와일드카드, 15년)
  3. 인증서 보관 — ~/.config/<project>-cert/origin.{crt,key}
  4. SSL 모드 — Cloudflare SSL/TLS → Full (strict)
  5. DNS — A 레코드 (apex + www + 필요한 서브도메인) → droplet IP, Proxied
  6. DigitalOcean droplet — Ubuntu 24.04, $6/월, SSH key 등록
  7. GHCR PAT — GitHub PAT (write:packages) → KAMAL_REGISTRY_PASSWORD ENV
  8. config/deploy.yml 의 TODO 항목 채우기 (image org, IP, hosts)
  9. 첫 배포 — bin/kamal setup
  10. 이후 — bin/kamal deploy
volumes 설정 안 하면 매 배포마다 SQLite DB 가 날아갑니다. 스크립트가 자동으로 추가하지만, 수동 작성 시 잊지 말 것.

7. 자주 마주치는 함정

"Let's Encrypt 발급 실패" — Cloudflare proxied 모드에서 ACME challenge 가 막힘. → Cloudflare Origin Cert 사용 (스크립트 템플릿 기본값).
"배포 후 DB 가 비어있음" — volumes 설정 누락. config/deploy.yml 에 "<name>_storage:/rails/storage" 있는지 확인.
"브라우저 에셋 404" — asset_path: /rails/public/assets 누락 시 무중단 배포 중 in-flight 요청이 구버전 에셋을 못 찾음.
"이미지 push 401 Unauthorized" — GitHub PAT 의 write:packages 스코프 누락. PAT 재발급 + KAMAL_REGISTRY_PASSWORD 환경변수 확인.
"config/master.key 가 커밋됨" — Rails 8 의 .gitignore 에 /config/*.key 가 있는지 확인. 이미 커밋했다면 즉시 키 재생성 + 히스토리 정리.
스크립트가 만든 .kamal/secrets 는 플레인텍스트 시크릿이 없습니다. 모든 값을 ENV 또는 외부 파일에서 읽도록 설계됨 — 그래서 git 커밋 안전.
