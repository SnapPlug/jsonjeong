DB 가 어디 저장되고, 어떻게 다루고, 어떻게 운영하는가 — 전체 라이프사이클

Stack: Rails 8 + SQLite + Active Record For: jsonjeong.com 및 모든 Rails 모놀리스
핵심 한 줄 — SQLite 의 본질은 "DB 가 디렉터리 안의 파일 한 개". 그래서 백업 = 파일 복사, 리셋 = 파일 삭제, 이관 = 파일 옮기기. Active Record 가 그 파일과의 모든 대화를 Ruby 객체로 추상화합니다.

1. 물리적 위치 — DB 가 어디 있나

SQLite 의 본질

DB = 디렉터리 안의 파일 하나.
앱과 같은 프로세스에서 파일 시스템 호출로 읽고 씀.

Postgres / MySQL 의 본질

DB = 별도 프로세스 (서버) + 그 프로세스가 관리하는 파일들.
앱이 5432 같은 포트로 네트워크 통신.

로컬 (내 맥)

~/projects/jsonjeong/
  └── storage/
      ├── development.sqlite3      ← 개발용 DB (실제 데이터)
      ├── development.sqlite3-shm  ← 공유 메모리 (트랜잭션)
      ├── development.sqlite3-wal  ← Write-Ahead Log (성능)
      ├── test.sqlite3             ← 테스트 실행 시 생성/삭제
      └── ...

프로덕션 (DigitalOcean droplet)

도커 컨테이너 안:
  /rails/storage/
      ├── production.sqlite3            ← 메인 DB
      ├── production_cache.sqlite3      ← Solid Cache
      ├── production_queue.sqlite3      ← Solid Queue
      └── production_cable.sqlite3      ← Solid Cable

도커 컨테이너 밖 (호스트 파일시스템):
  /var/lib/docker/volumes/jsonjeong_storage/_data/
      └── 같은 파일들이 영속 볼륨에 보관
컨테이너는 매 배포마다 새 것으로 교체되지만, 영속 볼륨이 호스트에 마운트돼서 DB 파일은 살아남습니다. config/deploy.yml 의 volumes: 설정이 결정적 — 빠뜨리면 매 배포마다 데이터 전부 날아감.

2. 환경별 DB 분리

Rails 는 자동으로 3개 환경의 DB 를 완전히 분리합니다.

Development
storage/development.sqlite3
로컬 개발 (bin/dev). 자유롭게 데이터 만들고 지움.
Test
storage/test.sqlite3
테스트 실행 시 (bin/rails test). 매번 자동 리셋.
Production
/rails/storage/production.sqlite3
배포된 서버. 영속 볼륨에 보관. 실제 사용자 데이터.

config/database.yml — Rails 8 기본값

default: &default
  adapter: sqlite3
  pool: <%= ENV.fetch("RAILS_MAX_THREADS") { 5 } %>
  timeout: 5000

development:
  <<: *default
  database: storage/development.sqlite3

test:
  <<: *default
  database: storage/test.sqlite3

production:
  primary: { <<: *default, database: storage/production.sqlite3 }
  cache:   { <<: *default, database: storage/production_cache.sqlite3 }
  queue:   { <<: *default, database: storage/production_queue.sqlite3 }
  cable:   { <<: *default, database: storage/production_cable.sqlite3 }
프로덕션엔 DB 파일이 4개 — 메인 + Solid Cache + Queue + Cable. 캐시가 폭증해도 메인 DB 가 안 느려집니다.

3. 코드에서 사용 — Active Record

SQL 을 직접 안 만집니다. Active Record (Rails 의 ORM) 가 객체로 추상화.

스키마 (db/schema.rb — 자동 생성)

ActiveRecord::Schema[8.1].define(version: 2026_05_13_030203) do
  create_table "subscribers", force: :cascade do |t|
    t.string   "email", null: false
    t.string   "source", default: "site", null: false
    t.string   "status", default: "pending", null: false
    t.datetime "beehiiv_synced_at"
    t.datetime "created_at", null: false
    t.datetime "updated_at", null: false
    t.index ["email"], unique: true
  end
end

모델 (app/models/subscriber.rb)

class Subscriber < ApplicationRecord
  # 그냥 클래스 하나. 컬럼 정의 안 써도 schema.rb 에서 자동 감지.
end

사용 — Ruby ↔ SQL

Ruby 코드 (Active Record)
Subscriber.create!(
  email: "x@y.com",
  source: "newsletter"
)

Subscriber.find_by(email: "x@y.com")

Subscriber.where(status: "pending").count

sub = Subscriber.find(1)
sub.update!(status: "active")

sub.destroy!
자동 번역된 SQL
INSERT INTO subscribers
  (email, source, ...)
  VALUES (...);

SELECT * FROM subscribers
  WHERE email = 'x@y.com' LIMIT 1;

SELECT COUNT(*) FROM subscribers
  WHERE status = 'pending';

UPDATE subscribers
  SET status = 'active', updated_at = ...
  WHERE id = 1;

DELETE FROM subscribers WHERE id = 1;

4. 스키마 변경 — 마이그레이션

DB 구조 바꿀 때 ALTER TABLE 직접 안 함. 마이그레이션 파일로 git 관리.

1
생성
bin/rails g migration AddPaidToSubscribers paid:boolean
→
2
파일 자동 생성
db/migrate/
20260520_*.rb
→
3
로컬 적용
bin/rails db:migrate
→
4
커밋
git commit
→
5
배포 시 자동 적용
bin/kamal deploy

마이그레이션 파일 예시

# db/migrate/20260520_add_paid_to_subscribers.rb
class AddPaidToSubscribers < ActiveRecord::Migration[8.1]
  def change
    add_column :subscribers, :paid, :boolean, default: false, null: false
  end
end

적용하면 자동으로 db/schema.rb 가 업데이트되고, 다음 SQL 이 실행됩니다:

ALTER TABLE subscribers ADD COLUMN paid BOOLEAN DEFAULT 0 NOT NULL;
되돌리기: bin/rails db:rollback — 가장 최근 마이그레이션 한 단계 취소.
핵심: 모든 팀원/서버가 같은 변경을 같은 순서로 적용. git 으로 관리되니까.

5. 운영 명령

로컬에서 데이터 확인

# Rails console (Ruby REPL)
bin/rails console
> Subscriber.count
> Subscriber.last
> Subscriber.where(status: "pending")

# 또는 SQLite CLI 로 직접
sqlite3 storage/development.sqlite3
sqlite> SELECT * FROM subscribers;
sqlite> .schema subscribers
sqlite> .tables
sqlite> .exit

프로덕션 데이터 확인

# 라이브 컨테이너에서 Rails console
bin/kamal app exec --interactive --reuse "bin/rails console"
> Subscriber.count

# 또는 컨테이너 bash 진입
bin/kamal shell
$ sqlite3 /rails/storage/production.sqlite3
sqlite> SELECT COUNT(*) FROM subscribers;

시드 데이터

# db/seeds.rb
User.create!(email_address: "admin@example.com", password: "...")

# 실행
bin/rails db:seed

DB 리셋 (개발 전용)

bin/rails db:drop db:create db:migrate db:seed
# storage/development.sqlite3 삭제 후 처음부터
프로덕션에서 절대 db:drop 금지! 모든 사용자 데이터가 즉시 사라집니다. 프로덕션 console 진입 시엔 destroy_all, delete_all 같은 명령도 신중하게.

6. 백업 전략

jsonjeong 의 현재 백업 전략: 없음 (TODO). droplet 디스크 사망 = 모든 데이터 손실. 사용자가 늘기 전에 반드시 구축할 것.

최소한의 수동 백업

# 프로덕션 DB 파일을 로컬로 scp
scp root@<droplet-IP>:/var/lib/docker/volumes/jsonjeong_storage/_data/production.sqlite3 \
    ~/backups/jsonjeong-$(date +%F).sqlite3

# 또는 Kamal shell 안에서 압축
bin/kamal shell
$ sqlite3 /rails/storage/production.sqlite3 ".backup /tmp/backup.sqlite3"
$ gzip /tmp/backup.sqlite3
cp 로 live 파일을 복사하면 WAL 모드라 손상 위험. SQLite 의 .backup 명령이 안전.

자동 백업 옵션

Litestream

SQLite 변경을 실시간으로 S3/R2 에 스트리밍 복제. 손실 거의 0.

비용: $0~ (R2 무료) 복잡도: 중 RPO: ≤ 1초

cron + scp

매일 droplet에서 SQLite .backup 후 로컬/S3 로 전송.

비용: $0 복잡도: 낮 RPO: 24시간

Solid Queue 잡

Rails 안에서 매일 잡으로 R2 업로드. Rails-네이티브.

비용: $0~ 복잡도: 중 RPO: 24시간

DO Snapshots

DigitalOcean droplet 전체 스냅샷. 클릭 1번이지만 비용 발생.

비용: ~$1/월 복잡도: 낮 RPO: 주 1회

7. 전체 라이프사이클 도식

   개발 (내 맥)                                프로덕션 (DO droplet)
   ─────────────                              ──────────────────

   bin/dev                                    bin/kamal deploy
     │                                          │
     ▼                                          ▼
   Rails 앱                                   Rails 앱 (docker container)
     │ Active Record                            │ Active Record
     ▼                                          ▼
   storage/development.sqlite3                /rails/storage/production.sqlite3
                                                │
                                                ▼ (volume mount)
                                              호스트 파일시스템 영속 볼륨
                                                │
                                                ▼ (백업 — TODO)
                                              Litestream → R2/S3


   마이그레이션 흐름:
   ──────────────
   bin/rails g migration AddX     →  db/migrate/*.rb 파일 생성
   bin/rails db:migrate           →  로컬 DB 에 ALTER TABLE 적용
   git commit                     →  마이그레이션 파일 커밋
   git push / bin/kamal deploy    →  서버에서 자동으로 db:migrate 실행
                                     (Kamal 의 prepared release hook)


   데이터 흐름:
   ──────────
   브라우저  →  Rails 컨트롤러  →  Active Record  →  SQLite 파일
                                       ↑
                       Ruby 객체 ←─────┘ (자동 변환)

8. SQLite vs Postgres / Supabase

관점SQLite (우리)Postgres (Supabase)
저장 형태 디렉터리 안 파일 1~4개 별도 서버 프로세스 + 데이터 디렉터리
앱과의 통신 같은 프로세스 (파일 시스템 호출, μs) TCP 5432 (네트워크, ms)
백업 단위 파일 복사 / .backup 명령 pg_dump / 매니지드 자동 백업
동시 쓰기 직렬화 (1개씩) — WAL 모드로 읽기는 무제한 진짜 동시 (MVCC)
확장 수직 (큰 디스크) 수평 (read replica, sharding)
운영 부담 0 — 그냥 파일 별도 인스턴스, 백업 설정, 버전 관리
비용 $0 (droplet 디스크에 같이) Supabase Free → Pro $25/월 + row/storage
이관 파일 복사 = 이관 완료 pg_dump → pg_restore (수십 분)
적합한 규모 1인 메이커 ~ 수만 동접 수십만+ 동접, 멀티 리전, 복잡 쿼리
단일 서버 / 수만 RPM 이하라면 SQLite 가 압도적으로 단순하고 빠릅니다. 37signals 의 Basecamp, marclou.com 등이 실제 SQLite 프로덕션 운영. 1인 메이커 서비스엔 정답. Postgres 가 필요해지는 시점은 한참 후 — premature optimization 회피.

한 줄 요약

SQLite + Active Record + 마이그레이션 + 영속 볼륨.
DB 가 "파일 하나" 라는 사실이 모든 운영을 단순하게 만든다 — 복사하면 백업, 삭제하면 리셋, 그게 다. 복잡도가 필요해질 때만 Postgres 로 이관하면 되고, 그날은 보통 영영 오지 않는다.
